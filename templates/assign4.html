Classes

Classes. Ah Classes. The backbone of the Objective-C framework, and some of the most oddly formatted pieces of foundation in any programming langauge. Classes are what make Objective-C what it is. They are how Objective-C is different from C as well as how it is different from C++. Objective-C and C++ implement classes in their own ways. In almost every program you write, you will create a custom class in order to encapsulate (ie wrap up) data and methods that do something with that data. They can seem somewhat daunting at first, but they become easier with practice. Classes let you define a blueprint for an object. They're similar to the C type struct. Once you have defined a class in a program, you can "instantiate" instances of the class to use in the program. For example, if I define a class Person that has the variables NSString *name, int age, NSArray *rightToeNames, and NSString *favoriteBahrainiOlympicHurdler, I can instantiate objects of that class that have those different attributes. One instance of that class could have a name of Joe and be age 19. Another instance of that class could have name Ryan and have rightToeNames be @[@"Jimmy", @"Maggie", @"Fred", @"Genghis", @"Frodo"]. The objects encapsulate the data stored in their properties.

Classes in Objective-C consist of two parts, an interface and an implementation. In C, prototypes are declared for functions at the top of the program so that the rest of the program is aware of their existence and can invoke them without getting compiling errors. These prototypes are also very helpful for programmers coming into a legacy codebase (existing code that they did not create but they need to use or make changes to it). They tell a programmer what functions exist, what they do (if you write good function names), what the function returns and what types of data you can pass to the function itself. In Objective-C, these prototypes are separated from their implementations into another file. That way, instead of having to read the lengthy implementation file with lots of cryptic code, a new developer can merely read the header file and know what functions already exist.

Objective-C classes have two main files. The interface of an Objective-C class is stored in a header file (.h suffix) and contains all the prototypes and variables for a class. These variables can be instance variables or properties (more on those later). The implementation for a class is stored in the implementation file (.m suffix). The implementation file contains all the definitions for the methods (name for functions that are part of a class) of the class. But more on zat laterz. Time for the examples.

Say I wanted to make a class Person like the one I mentioned above. I'll give it the above attributes as well as the method makeCryingNoise(). Note: I will define the attributes of the class using properties instead of instance variables (technically the properties are assigning values to the instance variables). You can read more about this here . It is always preferable to use properties instead of instance variables.
The first part of the file is the interface, which will contain properties and method prototype declarations.

// In Person.h
//

#import <Foundation/Foundation.h>

@interface Person : NSObject
// All Objective-C classes must inherit from another class. Almost always use NSObject.

@property NSString *name;
@property int age;
@property NSArray *rightToeNames;
@property NSString *favoriteBahrainiOlympicHurdler;
- (void) makeCryingNoise((NSString *)noise repetitionAmount:(int)repetitions);


@end

Objective-C method syntax:
minusOrPlusForClassOrInstanceMethod (returnType) functionName:(typeOfArgument1)argument1 secondArgumentSeparator:(typeOfArgument2)argument2
The above is all one line (see my actual method for an example)
Note a few things:
One, there is a minus or plus at the beginning.
Two, arguments are not specified inside () as in C. They follow a colon. To declare a method without arguments, you would do - (void) myMethod;
Three, multiple arguments are not separated by commas. They are separated by word separators specified by the method's creator (in this case, it's me).
The first argument doesn't have a word separator.

This would be the corresponding implementation (.m) file.

// In Person.m
//

#import <Foundation/Foundation.h>

@implementation Person

- (void) makeCryingNoise:(NSString *)noise repetitionAmount:(int)repetition
{
    for (int i = 0; i < repetition; i++)
    {
        NSLog(@"%@ cries %@", self.name noise);
    }
}

@end


Then, in my main file, I could import and use the class. I'll create a Person joe, set his name, age, right toe names, and favorite Bahraini Olympic hurdler.
// in main.m
//

#import <Foundation/Foundation.h>
#import "Person.h"

int main ()
{
     Person *joe = [[Person alloc] init];
    // Allocate memory from the class and initialize the variable (just do it)
    joe.name = @"Joe Moran";
    joe.age = 19;
    joe.rightToeNames = @[@"Hans von Aachen", @"Ernst Barlach", @"Johann Gottlieb Hantzsch", @"Vicco von Bulow", @"Fred"];
    joe.favoriteBahrainiOlympicHurdler = @"Ahmed Hamada Jassim";
    
    [joe makeCryingNoise:@"Deutsch cry" repetitionAmount:4];
    
    return 0;
    
}
